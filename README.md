# Meeting Room Server

一个用于学习网络编程的 C++ 会议室服务器。项目使用主进程 + 房间子进程 + 线程池的模型，支持创建会议、加入会议、房间内文本/图片/音频消息转发、关闭摄像头通知、成员加入/退出通知等功能。

## 功能

- TCP 长连接服务
- 多 accept 线程接收客户端连接
- 每个房间由一个子进程负责房间内事件循环
- 主进程维护房间状态、成员数量和业务房间号映射
- 通过 Unix domain socket + `SCM_RIGHTS` 在主进程和房间进程之间传递客户端 fd
- 房间进程内使用 `epoll` 监听成员连接
- 房间内使用发送队列和多个发送线程广播消息
- 自定义二进制帧协议，所有多字节整数字段使用网络字节序

## 编译

依赖：

- Linux
- g++，支持 C++17
- make
- pthread

编译：

```bash
make
```

清理：

```bash
make clean
```

## 运行

```bash
./server [port] [processes] [threads]
```

参数说明：

| 参数 | 说明 |
| --- | --- |
| `port` | 监听端口，范围 `0` 到 `65535` |
| `processes` | 房间子进程数量，也就是最多可同时占用的房间数量 |
| `threads` | 主进程 accept 线程数量 |

示例：

```bash
./server 8888 4 2
```

表示监听 `8888` 端口，预创建 `4` 个房间子进程，并启动 `2` 个 accept 线程。

## 架构

### 主进程

主进程负责：

- 创建监听 socket
- fork 房间子进程
- 启动 accept 线程
- 维护 `Manager` 房间管理器
- 监听房间子进程通过 pipe 发回的状态通知

主进程收到房间进程通知后会更新房间状态：

| 通知 | 含义 |
| --- | --- |
| `E` | 房主退出，房间释放 |
| `Q` | 普通成员退出，房间人数减一 |

### accept 线程

accept 线程接收新 TCP 连接后，先读取客户端的第一帧：

- `CREATE_MEETING`：创建新房间，将该连接作为房主 fd 传给对应房间进程
- `JOIN_MEETING`：按业务房间号加入已有房间，将该连接作为成员 fd 传给对应房间进程

创建房间成功后，服务器生成的业务房间号从 `100001` 开始递增。客户端看到的是业务房间号，不是内部数组下标。

### 房间进程

每个房间子进程维护一个 `room_manager`，负责：

- 保存房主 fd
- 保存成员 fd 集合
- 保存 fd 到客户端 IP 的映射
- 用 `epoll` 读取房间内消息
- 将待发送消息放入 `SEND_QUEUE`
- 由发送线程把消息广播给目标成员

房主退出时，房间会关闭所有成员连接并通知主进程释放房间。

## 协议帧

通用帧格式：

| 字段 | 大小 | 说明 |
| --- | --- | --- |
| start | 1 字节 | 固定为 `'$'` |
| type | 2 字节 | `MsgType`，网络字节序 |
| ip | 4 字节 | IPv4 地址字段，网络字节序；客户端请求中可填 `0` |
| payload_len | 4 字节 | payload 长度，网络字节序 |
| payload | 变长 | 业务数据 |
| end | 1 字节 | 固定为 `'#'` |

帧总长度为：

```text
1 + 2 + 4 + 4 + payload_len + 1
```

## 消息类型

当前 `MsgType` 定义如下：

| 值 | 名称 | 方向 | 说明 |
| --- | --- | --- | --- |
| `0` | `IMG_SEND` | 客户端 -> 服务器 | 发送图片数据 |
| `1` | `IMG_RECV` | 服务器 -> 客户端 | 转发图片数据 |
| `2` | `AUDIO_SEND` | 客户端 -> 服务器 | 发送音频数据 |
| `3` | `AUDIO_RECV` | 服务器 -> 客户端 | 转发音频数据 |
| `4` | `TEXT_SEND` | 客户端 -> 服务器 | 发送文本数据 |
| `5` | `TEXT_RECV` | 服务器 -> 客户端 | 转发文本数据 |
| `6` | `CREATE_MEETING` | 客户端 -> 服务器 | 创建会议 |
| `7` | `JOIN_MEETING` | 客户端 -> 服务器 | 加入会议 |
| `8` | `CLOSE_CAMERA` | 客户端 -> 服务器 -> 其他成员 | 关闭摄像头通知 |
| `9` | `EXIT_MEETING` | 客户端 -> 服务器 | 退出会议 |
| `20` | `CREATE_MEETING_RESPONSE` | 服务器 -> 客户端 | 创建会议响应 |
| `21` | `PARTNER_EXIT` | 服务器 -> 客户端 | 其他成员退出通知 |
| `22` | `PARTNER_JOIN` | 服务器 -> 客户端 | 新成员加入通知 |
| `23` | `JOIN_MEETING_RESPONSE` | 服务器 -> 客户端 | 加入会议响应 |
| `24` | `PARTNER_JOIN2` | 服务器 -> 客户端 | 加入成功后返回房间内已有成员 IP 列表 |

## 创建会议

客户端发送：

| 字段 | 值 |
| --- | --- |
| `type` | `CREATE_MEETING` (`6`) |
| `ip` | `0` |
| `payload_len` | `0` |
| `payload` | 空 |

成功后，房间进程向房主发送：

| 字段 | 值 |
| --- | --- |
| `type` | `CREATE_MEETING_RESPONSE` (`20`) |
| `payload_len` | `4` |
| `payload` | `uint32_t` 房间号，网络字节序 |

如果房间已满，响应 payload 中的房间号为 `0xffffffff`，对应服务端内部返回值 `-1`。

## 加入会议

客户端发送：

| 字段 | 值 |
| --- | --- |
| `type` | `JOIN_MEETING` (`7`) |
| `payload_len` | `4` |
| `payload` | `uint32_t` 房间号，网络字节序 |

主进程会先返回加入结果：

| 字段 | 值 |
| --- | --- |
| `type` | `JOIN_MEETING_RESPONSE` (`23`) |
| `payload_len` | `4` |
| `payload` | 成功为房间号，失败为 `0` |

加入成功后，连接 fd 会被传入对应房间进程，后续消息在房间进程内处理。

## 房间内消息

进入房间后，客户端可以发送：

- `TEXT_SEND`
- `IMG_SEND`
- `AUDIO_SEND`
- `CLOSE_CAMERA`
- `EXIT_MEETING`

`TEXT_SEND`、`IMG_SEND`、`AUDIO_SEND` 会被转换为对应的 `TEXT_RECV`、`IMG_RECV`、`AUDIO_RECV` 并广播给除发送者以外的其他成员。

`CLOSE_CAMERA` 要求 payload 为空，会广播给除发送者以外的其他成员。

`EXIT_MEETING` 要求 payload 为空：

- 如果退出者是房主，房间释放，所有成员连接关闭
- 如果退出者是普通成员，服务器通知其他成员 `PARTNER_EXIT`

## 成员通知

普通成员加入房间时：

- 其他成员收到 `PARTNER_JOIN`
- 新成员收到 `PARTNER_JOIN2`

`PARTNER_JOIN2` 的 payload 是已有成员 IPv4 地址列表，每个地址占 4 字节，均为网络字节序。

普通成员退出房间时：

- 其他成员收到 `PARTNER_EXIT`
- `ip` 字段为退出成员的 IPv4 地址

## 当前实现限制

- 房间最大成员数为 `50`
- 房间发送线程缓冲区大小为 `4MB`
- 当前协议没有鉴权、心跳、重连和消息持久化
- 当前主要面向学习和实验用途，客户端需要严格按协议完整发送帧
